# 问题定位与修复（含模型升级流程）

> **路径规则**：所有文档路径以 `smartroute.config.json` 中的 `documents` 配置为准，不要使用硬编码路径。

用户报告了一个问题：$ARGUMENTS

## 第一步：读取配置
1. 读取 `smartroute.config.json` 中 `documents` 配置，确定以下路径：
   - `system_test_cases`：系统测试用例文档路径
   - `unit_test_cases`：单元测试用例文档路径

## 第二阶段：定位（当前模型）
1. 分析问题描述，复现问题
2. 查看相关代码和日志，定位根因
3. 如果能定位到根因，跳到第三阶段
4. 如果经过多轮尝试仍无法定位，明确输出：
   "⚠️ 当前模型定位困难，建议切换到 opus 模型继续分析"
   并附上已排查的信息和思路

## 第三阶段：修复
1. 根据根因给出修改方案
2. 执行代码修改
3. 确保编译通过
4. 运行相关测试确认修复

## 第四阶段：漏测分析（需要 opus 模型）
问题修复后，必须执行以下分析：
1. 读取 `documents.system_test_cases` 和 `documents.unit_test_cases` 指定的文档
2. 判断该问题的复现场景是否已在测试例中
3. 如果不存在：将新场景作为新测试例追加到文档
4. 如果已存在：分析为什么会漏测，将原因记录在对应测试例的备注中
5. 保存更新后的测试例文档

## 第五阶段：回归测试（切回 minimax）
1. 遍历所有系统测试例，执行自动化测试
2. 运行所有单元测试
3. 确保无新增失败
4. 输出回归测试报告到 .pipeline/regression-report.md
